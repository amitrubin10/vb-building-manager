## 4. אלגוריתמים קריטיים

### 4.1 `getExpectedPayment(aptNum, year, month)`
מחזיר את המחיר לדירה בחודש ספציפי לפי התמחור. **אם התאריך לפני התקופה הראשונה — מחזיר 0.** פעם בטעות חישב המחיר של התקופה הראשונה גם לפני התקופה → תיקון חשוב.

### 4.2 `computeOBContribution(aptNum)` — לחישוב יתרת פתיחה
לוקח את החודשים הטרום-OB בטווח המסנן, מחשב:
- `pricingExpected` = תמחור × מספר חודשים
- `expected` = **max(pricingExpected, obDebt)** — אם OB > תמחור, יש "יתרה מועברת"
- `paid` = `expected - obDebt`
- `carryOver` = `max(0, obDebt - pricingExpected)`

**למה "max"?** כי אם OB גדול מהתמחור, יש חוב מ-תקופה עוד יותר ישנה. נכניס אותו ל"צפוי" כדי שהסה"כ יהיה מדויק.

### 4.3 `computeManualPaymentAllocation(payment, aptNum)` — לתשלום ידני יחיד
**עקרון FIFO — חוב ישן קודם:**
```
1. OB עודם (יתרת פתיחה היא הישנה ביותר) → obPaid
2. חודשים נבחרים לפי סדר כרונולוגי (ישן לחדש) → alloc
3. אחרון יכול להיות חלקי
4. יתרה נוספת (אם amount > OB + sum months) → overflow (general credit)
```

**מחזיר:** `{alloc, overflow, obPaid}`

### 4.3.5 `applyFifoReallocation()` — FIFO חוצה-מקורות (חדש, 2026-04-24)
**הקריטי ביותר — רץ בסוף `analyzeCollection` לכל דירה.**

**העיקרון:** **כל כסף נכנס** (בנק / אשראי / ידני כללי / overflow+obPaid מתשלום ידני) — הולך FIFO לכל החובות:
```
loosePool = sum(bankCreditPaid per month) + getManualPaymentsTotal
           (מינוס earmarks של bank-alloc marks, כדי לא לספור פעמיים)

1. OB → obCredit = min(obDebt, loosePool)
2. כל חודש (מהישן): applied = min(deficit, loosePool)
3. יתרה → window._fifoByApt[aptNum].overflow (קרדיט זמין ברמת דירה)
```

**מכבד בחירות משתמש מפורשות:**
- תשלום ידני עם חודשים ספציפיים → `_fixedMp` נשאר באותם חודשים
- סימוני חודש (`marked` / `bank-alloc`) → `_markCovers = expected` נשאר באותם חודשים
- רק הכסף ה"חופשי" (בנק + manual loose) מופעל בFIFO

**מעדכן את כל `rec`:**
- `rec.paid = _fixedTotal + _fifoApplied` (מוגבל ב-expected — אין יותר "שילם ביתר" מתנועת בנק שהופנתה אחורה)
- `rec.status` מחושב מחדש
- `rec._redirected` = כמה כסף של תנועות בנק בחודש הזה הופנה אחורה (לUI)

**מאוחסן ב-`window._fifoByApt[aptNum]`:**
- `obCredit` — כמה הlooose pool סגר מ-OB
- `obDebt` — חוב OB מקורי
- `overflow` — קרדיט נותר (ברמת דירה)

**צרכנים:**
- `renderCollectionTable` (סיכום הדירה): `totalPaid += obCredit + overflow`
- `buildApartmentView` (OB row + summary + debtByYear + קרדיט זמין banner)
- `addManualPayment` (באנר "חוב נוכחי" — משתמש ב-FIFO-adjusted debt)

### 4.4 `getManualPaymentsTotal(aptNum)`
**מכליל 3 מקורות של "קרדיט":**
- תשלומים ידניים ללא חודשים (general credit)
- `obPaid` מתשלומים ספציפיים-חודש
- `overflow` מתשלומים ספציפיים-חודש

### 4.5 ביטול אוטומטי בבחירת חודשים (דינמי)

**ב-`updateMpAllocationPreview` (תצוגה מקדימה חיה):**
- **מקטינים סכום** → מורידים סימון מחודשים שמקבלים 0
- **מגדילים סכום** → מוסיפים חודשים חייבים (ואז עתידיים) לפי סדר כרונולוגי
- **אם הסכום ≤ OB** → מעבר אוטומטי למצב "קרדיט כללי"
- **אם הסכום > OB ויש חודשים חייבים ובמצב "קרדיט"** → מעבר אוטומטי ל"חודשים ספציפיים"
- **יתרה קטנה לסוף** → חודש נוסף מתווסף כתשלום חלקי (לא overflow)

### 4.6 `computePropertyTotals(property)` — עבור נכסים מושכרים
אותה לוגיקה כמו דירות, עם OB משלו. מפצל בין pre-OB ו-post-OB. מזהה תנועות בנק לפי **keywords**.

## 5. היסטוריית החלטות קריטיות

### UX של יתרת פתיחה
- **לא אוטומטית** מקובץ האקסל לנכסים — רק לדירות. החלטה: למנוע בלבול בין שתי המהויות.
- **כפתור "OB=0 לכל הלא-מופיעים"** בקליטה — לסמן דירות ששילמו כסדרן.

### תשלום ידני — 3 מצבים
- **קרדיט כללי** — מפחית מסה"כ חוב
- **חודשים ספציפיים** — מסמן חודשים כשולמים עם פיצול
- **יתרה אוטומטית** — overflow → היה "קרדיט כללי", שונה לחודש נוסף partial

### סימון חודש כשולם — 3 אפשרויות
1. שיוך לתנועה בנקאית קיימת (לתרחיש "5000 באפריל מכסה כמה חודשים")
2. תשלום ידני חדש (כמו "מזומן עבור מאי")
3. סימון בלבד (בלי סכום — רק להסיר חוב)

### סדר העדיפויות בתשלום
**FIFO — חוב ישן קודם.** זה עקרון חשבונאי נכון. דיון ארוך עם המשתמש הסתיים בהחלטה הזו.

### תצוגת כרטסת — שלב 1 (2026-04-24)
**מטרה:** תצוגה אלטרנטיבית בכרטיס הדירה, לקריאה בלבד, עם **חיובים וזיכויים ככרטסת חשבונאית**.

**מיקום:** כפתור מתג בראש ה-drilldownBody — `📅 חודשית` (default) / `📋 כרטסת`.
**state:** `_drilldownMode` — נאפס ל-'monthly' בכל פתיחה טרייה. `switchDrilldownMode(mode)` מחליף ומבצע re-render.

**מבנה נתונים (`collectLedgerEntries`):**
- חיובים:
  - `preOB-month` — שורה לכל חודש טרום-OB בפילטר (אופציה ב' של המשתמש)
  - `month` — שורה לכל חודש פעיל
- זיכויים:
  - `obImplied` — הסכום ששולם טרום-OB (obContrib.paid), מתוארך ל-openingBalanceDate, מוגבל ל-preOB charges בלבד
  - `bank` / `credit-card` — תנועות מ-allIncomeData מסוננות לפי allactiveMonths
  - `manual` — כל התשלומים הידניים

**התאמה (`matchLedgerEntries`) — 3 passes:**
1. **Pinned** — `manual + months[]`: משתמש ב-`computeManualPaymentAllocation`: obPaid→preOB FIFO, alloc→months ספציפיים, overflow נשאר loose.
2. **OB implied** — FIFO אבל **רק בתוך preOB charges** (לא דולף לחודשים פעילים).
3. **Loose** — כל השאר, מסודר כרונולוגית, FIFO לחוב הישן ביותר הפתוח.

**עמודות טבלה:** תאריך · תיאור · חיוב · זיכוי · יתרה (running) · התאמה (list of links to charges/credits).
**סיכום למטה:** סה"כ חיובים, זיכויים, יתרת חוב פתוחה, קרדיט לא מנוצל.

**מגבלות ידועות בשלב 1:**
- marks (`marked` / `bank-alloc`) לא מיוצגים בכרטסת — הFIFO שלה עצמאי. המשתמשים שמשתמשים במנגנון המיושן הזה יראו סדר התאמה מעט שונה מהתצוגה החודשית (אבל סה"כ החוב זהה).
- לחיצה על שורה לא עושה כלום (קריאה בלבד) — מתוכנן לשלב 2.

**קבצים שנגעו בהם:**
- `let _drilldownMode = 'monthly';` — מצב
- `switchDrilldownMode(mode)` — החלפה
- `openAptDrilldown(aptNum, preserveMode?)` — סיגנטורה התעדכנה
- טוגל HTML בתחילת `tableHtml`
- `collectLedgerEntries`, `matchLedgerEntries`, `renderLedgerView` — פונקציות חדשות אחרי `openAptDrilldown`

### FIFO חוצה-מקורות (2026-04-24)
**שינוי משמעותי:** קודם FIFO חל רק על תשלום ידני. עכשיו **גם על תנועות בנק ואשראי**.
- **לפני:** תנועת בנק של 1,130 באפריל → "שילם ביתר על אפריל", OB נשאר פתוח.
- **אחרי:** תנועת בנק של 1,130 באפריל → סוגרת OB אוטומטית, אפריל נשאר פתוח (565).
- **רציונל:** הדייר התכוון לסגור את החוב, לא לשלם ביתר. זה גם הסטנדרט החשבונאי.
- **UI:** בשורת החודש מופיע "💡 X הופנו לסגירת חוב ישן יותר קודם (FIFO)" ובשורת OB — "+ שולם מתנועות בנק/אשראי: X (נסגר ✓)".
- **קרדיט זמין:** כשכל החובות נסגרו ונשאר עודף → באנר ירוק בראש הדריל-דאון "💰 קרדיט זמין: X₪".
- **bank-alloc marks:** כבר לא נחוצים (FIFO מפזר אוטומטית). מתוחזקים לתאימות אחורה ומחייבים earmark מהpool.

### "יתרה מועברת" בשורת OB
- **גישה מתמטית** (OB > תמחור) — עבדה אבל הייתה לא אינטואיטיבית
- **גישה סופית** (לפי פיצול y_N) — כל שנה לפני תחילת המסנן מסומנת ב-⚠️ צהוב

### ברירות מחדל חשובות
- **מסנן תאריכים:** ינואר של השנה הנוכחית עד החודש הנוכחי
- **מצב תשלום:** לא מסומן "הסתר דירות המשויכות לסוגים אחרים"
- **לשונית יתרות פתיחה:** ברירת מחדל = הצגה רק של דירות עם חוב > 0

### מיזוג קבצים (Dedup)
- **ברירת מחדל:** מיזוג + דילוג על כפילויות
- **מפתח לכפילות בנק:** `תאריך + סכום + אסמכתה`. אם אין אסמכתה — אישור מהמשתמש והעברה ל-`תאריך + סכום + תיאור`
- **מפתח באשראי Hombi:** `שנה + חודש + דירה` → דריסה (עדכון)

### UI/UX החלטות
- **"הגדרות תמחור" שונה ל"הגדרות הבניין"** — מודגש עם אייקון SVG של בניין
- **שילוב טאב "שיוך דירות"** — עבר להיות **כפתור "רשימת דירות"** בתוך כל סוג דירה
- **לשוניות הגדרות הבניין:** 4 — סוגי דירות/יתרות פתיחה/נכסים מושכרים/גיבוי
- **כפתור "+ תשלום ידני"** עבר לכרטיס נפרד ליד "יצירת קשר" בכרטיס דירה
- **סדר מסך חדש:** טבלת גבייה → KPIs → גרפים → הוצאות → תנועות

## 6. פיצ'רים עיקריים (רשימה מקיפה)

### הגדרות הבניין (4 לשוניות)
- **סוגי דירות ומחירים** — grid של כרטיסים, כל סוג עם תקופות מחיר ושיוך דירות
- **יתרות פתיחה** — קליטה מאקסל (עם בחירת גיליון + מיפוי עמודות), סינון "רק עם חוב", OB=0 אוטומטי
- **נכסים מושכרים** — רשימה + עריכה (ולא אחד מתחת לשני)
- **גיבוי ושחזור** — ייצוא/ייבוא JSON בצד ימין ושמאל

### מסננים
- **לוח שנה native** (input type=month) עם כפתור ניקוי X
- **טווח:** 24 חודשים אחורה עד 24 חודשים קדימה
- **ברירת מחדל:** תחילת השנה הנוכחית עד היום

## 11. תוכנית הכרטסת — שלבי פיתוח 📋

הכרטסת היא תצוגה אלטרנטיבית בכרטיס דירה, בסגנון "דף חשבון בנק". מתפתחת ב-4 שלבים.

### שלב 1 — תצוגה לקריאה ✅ הושלם (2026-04-30)
ראה סעיף 4.3.5 ו-5 לפרטים מלאים. תמצית:
- כפתור מתג בראש כרטיס דירה: `📅 חודשית` (default) / `📋 כרטסת`
- טבלה כרונולוגית: חיוב | זיכוי | יתרה רצה | עמודת התאמה
- FIFO מחושב בכרטסת באופן עצמאי (לא דרך `applyFifoReallocation`) — לכן יכולה להופיע סטייה קטנה מהתצוגה החודשית כשיש marks ישנים, אבל הסה"כ זהה.

### שלב 2 — עקיפה ידנית של זיכויים (NEXT — הצעד הבא הקרוב)

**מטרה:** משתמש שמכיר את הכוונה האמיתית של הדייר (שונה מ-FIFO האוטומטי) — יוכל לעקוף ידנית.

**UI מתוכנן:**
- לחיצה על שורת זיכוי בכרטסת → פופאפ "לאן הולך התשלום?":
  - מצב ברירת מחדל מסומן: `FIFO אוטומטי` (כפי שמחושב היום)
  - אפשרויות עקיפה: רדיו לחיוב יחיד / inputs לפיצול ידני בין כמה חיובים
  - אם עקיפה כבר פעילה: כפתור בולט `🔄 חזור ל-FIFO אוטומטי` בראש הפופאפ
- **תג ויזואלי 🔒** על שורות זיכוי עם עקיפה פעילה — לסקירה מהירה
- **תג `ⓘ` עם hover** ליד כל ✓ ירוק (התאמה) — מציג מקור התשלום ותאריך

**אחסון חדש מתוכנן:** `vaad_credit_overrides` ב-localStorage:
```js
{
  [aptNum]: {
    [creditId]: {
      allocations: [{chargeId: "chrg_month_2026_4", amount: 565}, ...]
    }
  }
}
```

**שילוב עם FIFO:**
- ב-`matchLedgerEntries`: לפני pass 3 (loose FIFO), Pass 0 חדש שקורא overrides ומיישם אותם
- מחיקת אוברד ספציפי כשמשתמש לוחץ "חזור ל-FIFO"
- האם להעביר את האוברד גם ל-`applyFifoReallocation` (הלוגיקה הראשית)? **לדיון** — אם כן, ההשפעה תורגש גם בתצוגה החודשית. אם לא, האוברד הוא רק ויזואלי בכרטסת.

**שאלות פתוחות:** איך מתמודדים עם תאריכים פגי תוקף (זיכוי שכבר נסגר, ובא משתמש לעקוף עליו)?

### שלב 3 — שיוך דייר מחדש + "התעלם" (אחרי שלב 2)

**מטרה:** תנועה בבנק שזוהתה לדייר לא נכון (matchTransactionToTenant החזיר תוצאה שגויה) — אפשרות לתקן או להתעלם.

**UI מתוכנן:**
- **באותו פופאפ** של שלב 2 — תוספת חלק "שינוי שיוך":
  - רדיו 1: השאר בדירה X (default)
  - רדיו 2: העבר לדייר: `[▼ dropdown של כל הדירות]`
  - רדיו 3: סמן כלא קשור לגבייה (התעלם — לא ייספר באף דירה)
- **תג ⚠️ אוטומטי** על תנועות עם confidence < 70% — מזמין בדיקה (בלי לכפות)

**אחסון חדש:** `vaad_tx_reassignments`:
```js
{
  [txKey]: {  // txKey = date + amount + ref
    aptNum: "X" | null  // null = ignored
  }
}
```

`matchTransactionToTenant` תקרא ראשון מהאוברד; אם יש — תחזיר את הדייר/null שצוין. אחרת — לוגיקת זיהוי רגילה.

### החלטות UX לכרטסת (לפני שלבים 2-3)

המשתמש בחר במפורש את הגישות הבאות. **חשוב לכבד אותן בזמן הפיתוח של שלבים 2-3:**

**שאלה 1 — איך מבטלים עקיפה ידנית?**
- ✅ **אופציה ב' (פופאפ)** — קליק על שורת זיכוי פותח פופאפ, בו כפתור בולט "🔄 חזור ל-FIFO"
- + **תג ויזואלי 🔒** על שורות עם עקיפה פעילה (לסקירה)
- **רציונל:** עקביות עם דפוס "click row → popup". פחות רעש ויזואלי. רואים את ההקשר לפני ביטול.

**שאלה 2 — תשלום אחד שחוצה כמה חיובים: אישור או שקט?**
- ✅ **אופציה א' (שקט + חיווי ויזואלי)** — פיצול אוטומטי ללא פופאפ
- + **רמז `ⓘ` עם hover** ליד כל ✓ ירוק (התאמה) — מציג מקור התשלום
- **רציונל:** פיצול הוא התנהגות צפויה (FIFO), לא חריגה. אישור על כל פעם → המשתמש יתעייף ויאשר אוטומטית. החיווי הוויזואלי כבר פותר את השקיפות.

**שאלה 3 — תנועות בנק שגויות: איך מתקנים?**
- ✅ **אופציה ג' (פופאפ אחד עם dropdown)** — אותו פופאפ של עקיפה, עם תוספת לשינוי שיוך
- אופציות: השאר / העבר לדייר אחר / סמן כלא קשור (התעלם)
- + **תג ⚠️ אוטומטי** על תנועות עם confidence < 70%
- **רציונל:** עקביות (אותו דפוס לכל פעולה על שורה). גמישות לכל המקרים. לא דורש זיהוי מראש מה הבעיה.

### החלטה מפתח: כרטסת היא **תוספת**, לא החלפה

המשתמש שקל להחליף את התצוגה החודשית בכרטסת לחלוטין. **אחרי דיון נבחרה גישה היברידית:**

- כרטסת = "לחיצה אחת במרחק" — לחקירה מעמיקה / דיווחים / משתמשים מתקדמים
- שתי התצוגות מציגות **אותם נתונים**, רק בפורמטים שונים. אסור לחישוב להיות שונה.
